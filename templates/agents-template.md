# AGENTS.md

<项目名> Agent 协作说明。

## 通用要求

- 使用中文沟通和写项目文档。
- 开始任务前先读取 `README.md`、`CLAUDE.md` 和相关 `docs/`。
- 新需求、Bug 修复、方案设计、接口变更、数据模型变更或核心 UI 变更开始前，必须先阅读：
  - `docs/system/current-architecture.md`
  - `docs/system/product-rules.md`
  - `docs/system/data-model.md`
  - `docs/system/api-inventory.md`
  - `docs/system/known-pitfalls.md`
- 必须搜索相关代码和测试，确认当前系统事实后再写方案或改代码。
- 不能凭猜测改数据库字段、文件字段、接口语义或核心业务规则。
- Bug 修复必须说明根因、影响范围和回归测试。
- 修改接口、数据模型、核心产品规则或用户已确认交互时，必须同步更新 `docs/system/`。
- 只做最小可行改动，不做无关重构。
- 不删除用户未明确要求删除的能力。
- 不回滚用户或其他 Agent 的改动。

## 新需求开工前必答

任何新需求开始前，Agent 必须回答：

- 这个需求影响哪些已有模块？
- 当前系统是否已有类似能力？
- 是否和产品规则冲突？
- 是否和数据模型字段语义冲突？
- 是否会重新引入 `known-pitfalls.md` 中的问题？
- 是否影响本地数据或线上服务？
- 最小可行改动是什么？
- 需要补哪些测试防止回归？

如果答不清楚，必须继续查代码、查文档、查测试，或者向用户提问，不能猜。

## 不允许的行为

- 不把研究型方法直接宣称为生产可用。
- 不在没有验证的情况下承诺商用质量。
- 不绕过 License 处理第三方依赖。
- 不写入密钥、token 或隐私数据到文档中。

## 验收口径

任何阶段都要给出可验证产物：

- 输入数据
- 处理命令或任务记录
- 输出结果
- 关键指标（耗时、大小、成功率等）
- 失败原因和下一步修复建议
